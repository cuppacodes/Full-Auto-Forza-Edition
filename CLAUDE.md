# Full Auto Forza Edition (FAFE) — Project Summary

## What is this?
A Windows desktop automation tool for Forza Horizon 6. Built with Python + CustomTkinter. Uses a hybrid OpenCV + optional OCR detection pipeline to identify game screens and simulate keyboard/mouse input.

## Architecture

### UI (main_window.py)
- **Fluent sidebar layout** (`_build_ui` → `body` grid: col 0 sidebar fixed 210px, col 1 main; status bar packed at the window bottom, full width). **Sidebar** (`_build_sidebar`) is a **rounded bordered card** (`sidebar_bg` + `border`): FAFE wordmark with an **accent underline**, the 4 emoji-free function nav items (`_nav_buttons` keyed `race`/`mastery`/`buy`/`delete`, labels from `nav_*` strings), a spacer, then a bottom group — a compact 🐞/Discord row, **Settings** (`_nav_buttons["settings"]`), and **Support Me** (an **outlined** accent button). **Active nav** = subtle `surface_alt` fill + bright text + an **accent left bar** (`_nav_bars`, driven by `_set_nav_active`); inactive = transparent + `text_muted` + a bar blended into `sidebar_bg`. **Main column** (`_build_header`): header row with the **page title** (`_page_title_var`, per-tab from `page_title_*`) left + **monitor picker** + **overlay on/off indicator** right; below it `_main_content` holds the per-tab content frames AND the inline panels. The Setup panel, the Start/Stop row, and the log are each **bordered cards**; the log (`LogWidget`) shows an **"Activity"** header + separator and colors `--`-prefixed loop headers with `log_accent`.
- **One main view at a time (`_show_main`)**: the 4 tab frames and the settings/support/report panels are ALL children of `_main_content`, shown one at a time via `_show_main(frame)` (pack_forget the previous, pack the next). `_switch_tab(tab)` calls it, sets `_current_tab` / `_active_log` / `_active_setup_panel`, highlights the active nav via `_set_nav_active` (accent fill), and updates the page title. `_open_settings`/`_open_support`/`_open_report_help` show their panel via `_show_main` (sidebar stays visible; Settings nav highlights); their `_close_*` just calls `_switch_tab(_current_tab)`. **Tabs stay clickable while a panel is open** — clicking one exits the panel (this also fixed the old crash where switching mode mid-settings broke the app). No early-return in `_switch_tab` (a panel may be open with `_current_tab` still set).
- Each tab has: description label, optional count input (0=unlimited), Start/Stop buttons, log widget
- Auto Unlock 22B tab also has a start-row selector (CTkSegmentedButton, values 1/2/3) between count and run controls; saved as `mastery_start_loop` in config
- **Support panel** (`_build_support_panel`, parented to `_main_content`): two tip methods — a **街口支付 (JKOPAY) QR** (260×260) and a **PayPal** button sized to the QR width. Discord/PayPal/QR images load via `_load_ctk_image` (PIL→CTkImage) resolved by `_resource_path` (dev + frozen `--onedir`/`_MEIPASS`), graceful fallback if missing. Assets bundled via `--add-data "assets;assets"` (`support_jkopay.png`, `discord_logo.png`, `paypal_logo.png`).
- Language: zh-tw (default), en — switching triggers app restart. (Simplified Chinese zh-cn / `chs` templates were removed; `config.load` migrates any stored `lang=zh-cn` → `zh-tw` and `template_lang=chs` → `auto`.)
- **Theme presets** (replaces the old light/dark switch + accent picker): `theme.THEME_PRESETS` (token dicts) — `"default"` (current look, leaves CTk defaults alone) and `"midnight_neon"` (navy/cyan, restyles CTk's ThemeManager). Picked in Settings → Appearance (`theme_preset` config); changing restarts the app. `MainWindow._t(key)` looks up a token; `theme.apply_preset` (called in `_apply_theme`, after `set_default_color_theme`) pushes non-default colors into CTk + reassigns `theme.START_FG`/`STOP_FG`/etc. so existing widgets theme automatically. Fonts in tokens come from the locked CJK-safe font tokens. See theme.py.
- **App icon**: `FAFE_icon.ico` is both the exe icon (PyInstaller `--icon`) and the runtime window/taskbar icon. `--icon` only covers the exe; the window icon is set by `MainWindow._set_window_icon()` via `iconbitmap`. CustomTkinter applies its own default icon ~200ms after init, so we re-assert ours on a `self.after(300, …)` delay or it gets overridden. `_icon_path()` finds the `.ico` in both dev (next to source) and frozen `--onedir` (`sys._MEIPASS` / next to exe); the icon is bundled via `--add-data "FAFE_icon.ico;."` so it exists at runtime.
- **Fonts — locked per language, 3 layers** (`theme.init_fonts(lang, root)` + `theme.apply_font_family_to_tree(root)`): the UI font family is pinned to a guaranteed-installed Windows face per language — **繁中 → `Microsoft JhengHei UI`**, **简中 → `Microsoft YaHei UI`**, **en/other → `Segoe UI`** (each with installed-font + deterministic fallbacks via `_first_installed`). Why: `Segoe UI`, hardcoded `Arial`/`Segoe UI` font tuples, *and* CustomTkinter's default `Roboto` all have **no CJK glyphs**, so drawing Chinese makes Windows fall back through its font-linking registry — and on some users' machines that fallback resolves to a **decorative/brush font** (a real user report). Naming a CJK-capable font explicitly bypasses font-linking. Three widget-font paths all had to be covered:
  1. **`theme.*_FONT` tokens** — `init_fonts` rebuilds every token (all callers use `theme.X_FONT` attribute access — never `from theme import X_FONT` — so reassignment propagates) and sets Tk named fonts (`TkDefaultFont`/`TkTextFont`/`TkMenuFont`/…, but **not** `TkFixedFont`, to preserve monospace).
  2. **CTk widgets with `font=None`** (e.g. the tab buttons) — `init_fonts` overrides `ctk.ThemeManager.theme["CTkFont"]["family"]`. **CRITICAL ORDERING**: `init_fonts` MUST run **after** `_apply_theme()` — `set_default_color_theme()` reloads the theme JSON and resets `CTkFont.family` back to `Roboto`, so calling `init_fonts` before it gets clobbered (this was the "body text fixed but buttons still brush-font" bug). Order in `__init__`: `super().__init__()` → `_apply_theme()` → **`init_fonts()`** → `_build_ui()`.
  3. **Widgets with a hardcoded `('Arial'/'Segoe UI', N)` tuple** (dozens across `main_window.py`/`setup_panel.py`: tabs, dropdowns, segmented buttons, labels) — these bypass both 1 and 2. `apply_font_family_to_tree(root)` walks the whole widget tree post-build and rewrites each widget's `font` **and** CTkOptionMenu/CTkComboBox's separate **`dropdown_font`** to the locked family (preserving size/weight), handling `CTkFont` objects, tuples, and bare strings. Idempotent. Called via `MainWindow._apply_ui_fonts()` at the end of `_build_ui` and after each on-demand panel build (settings/support/report-help). Dynamic toplevels created later are covered too: the tooltip uses `theme.UI_FAMILY` inline; the update dialog and example-image window each call `apply_font_family_to_tree(win)` after building.

  `init_fonts` must run **after** the Tk root exists (it queries `tkinter.font.families()`). `overlay.py` (plain Tk) reads the resolved family via its `_ui_family()` helper. Changing language restarts the app, so the family is re-resolved on next launch.
- **UI scale** (Settings → Appearance, `ui_scale` config): `Auto` / `100%`–`250%`. Drives `ctk.set_widget_scaling` + `set_window_scaling` from a single factor resolved by `config.resolve_ui_scale` (Auto = `get_scale_factor()` = screen_height/1080, clamped 0.8–2.5). `MainWindow._apply_scaling()` runs **before `super().__init__()`** (before the Tk root exists): it marks the process DPI-aware (crisp text) and calls `ctk.deactivate_automatic_dpi_awareness()` so scaling is driven only by our factor, not stacked on CTk's per-monitor DPI — so a given % means the same size regardless of the user's Windows display-scaling. Changing it applies live (`_on_scale_change` re-sets scaling + re-applies base geometry `900x650` so the window refits). **Gotcha**: CTk multiplies `CTkImage` sizes *and* `CTkToplevel.geometry()` by this factor, so anything sized from raw screen pixels must divide by it — e.g. `_show_example_win` computes its fit budget as `screen*0.80 / _ui_scale`, else the example image renders ~scale× oversized.

### Detection (detector.py)
- `ScreenDetector.detect(frame, key, template, threshold)` returns a `MatchResult` dataclass
- ROI-first: each template key has a configured search region (see `DEFAULT_ROIS`); full-screen fallback applies a 0.94 score penalty
- **Ultrawide / non-16:9 ROIs — per-element anchor** (`_roi_for_frame(key, roi, w, h)`): Forza anchors UI two ways on ultrawide, so a single rule doesn't work: **menus/dialogs** sit in a **centred 16:9 box**, while **in-game HUD** (the race timer `racing`/時間, restart prompt, etc.) anchors to the **screen edges**, outside that box. So keys in **`_ROI_BOX_KEYS`** (currently just **`start_menu`**) map their ROI *into the centred box* (tight — `start_menu` false-matched the loading screen at ~85% on a wide band, causing stuck/Enter-spam on loop 2+); **every other key uses a full-axis band** (full-width on ultrawide / full-height on tall) so edge-anchored HUD is still found. History: full-width-band-for-all → start_menu false-matched; centred-box-for-all → fixed start_menu but BROKE `racing` (HUD is at the edge, outside the box); per-key is the resolution. 16:9 (within `detector_roi_aspect_tol`) is untouched. Toggle off via `detector_roi_aspect_fix=false`. Add a key to `_ROI_BOX_KEYS` only if it's a centred menu that false-matches on the band. Mastery node *click* positions use the centred-box model in `load_nodes` (see Capture).
- **Text template fast path**: `TEXT_TEMPLATES = frozenset(DEFAULT_ROIS.keys())` — all 11 keys. Text templates skip the Canny edge channel and use only 3 scales (`detector_text_scales`, default `[0.95, 1.00, 1.05]`), reducing matchTemplate calls from 14 to 3 (~4–5× faster). Non-text templates keep the full 7-scale + edge pipeline.
- Full pipeline scales: `[0.86, 0.92, 0.97, 1.00, 1.03, 1.08, 1.14]`
- Hybrid score (non-text): `0.62 × grayscale(equalizeHist) + 0.28 × Canny edges + up to 0.10 OCR bonus`
- Text score: `grayscale(equalizeHist)` only (edge channel skipped)
- **Default mode** (`detector_ocr_primary: true`, default): For text templates with OCR hints, OCR is a first-class **confirmation** signal — not a +0.10 bonus. Pixel template matching of game UI text is fragile across hardware (GPU AA, font hinting, HDR, game settings all change pixels) — text content is what's actually invariant, so finding the hint text promotes a weak pixel match to a confident one. Logic (decision band `detector_ocr_skip_below` ≤ score < `detector_ocr_skip_above`): if `image_score >= detector_ocr_skip_above` (default 0.90), trust the pixel match, skip OCR; if `image_score < detector_ocr_skip_below` (default 0.20), too low to be real, skip OCR; **in between**, run OCR (or use a cached confirm) and if the hint is present promote to `max(image_score, detector_ocr_confirm_score)` (default 0.85). A confirmation is **cached** per key for `detector_ocr_cache_duration` (gated by `detector_ocr_cache_pixel_min`, default 0.15) so the per-key OCR cooldown doesn't break the stability filter. **If no OCR backend is installed** the gate is skipped (pixel-only), so detection still works.
  - **OCR VETO — opt-in, `detector_ocr_veto` default `false`.** *Why it's not a hard default:* a small text template scanned multi-scale over a large ROI can hit a high `TM_CCOEFF` (~0.80) on an unrelated scene (a race scene false-matched `restart_menu`/`confirm` at ~80%), and on hardware where genuine matches *also* only score ~0.75–0.80 pixels can't separate them. The tempting fix — *reject* any match the hint doesn't confirm — was tried as the default and **regressed hard**: on machines where OCR can't reliably read the game text, every genuine screen got vetoed and the script **stalled on every step**. So by default OCR can only **confirm**, never block; a screen OCR can't confirm keeps its pixel score (original working behaviour). When `detector_ocr_veto=true`, the veto fires **only on a positive contradiction** — OCR actually read text (`ocr_text` non-empty) that doesn't contain the hint → cap to `detector_ocr_veto_ceiling` (default 0.40, below the 0.45 soft-threshold floor). It **never** vetoes a silent/failed OCR, so even enabled it won't stall a machine whose OCR reads nothing. The **dominant** ghost cause (the overlay being captured and self-detected) is handled separately and unconditionally by the overlay capture-exclusion (see Game status overlay), so most users never need the veto. Uses tight 3-scale (`detector_text_scales`) gray-only matching with ROI-first + 0.94-penalty full-screen fallback.
- **Detection mode is automatic (no manual toggle)**: `detector_ocr_primary` is set per run from the template type — **preset resolution (1080p/1440p/2160p) → Default mode (`true`)**, **custom templates → Custom mode (`false`)** — in `race.run` / `mastery.run` right after `res` is resolved (`_fresh['detector_ocr_primary'] = (res != 'custom')`). The old topbar Default/Custom segmented button was removed (most users found it confusing); any `detector_ocr_primary` value in config.json is overridden at run start by this rule.
- **Custom mode** (`detector_ocr_primary: false`, auto-selected when the chosen resolution is `custom`): For users with custom-captured templates that differ from the defaults (different language, non-text content like icons, different in-game UI position, captured at non-standard scale). Custom mode bypasses every Default-mode assumption: (1) **OCR fully disabled** — no first-class confirmation, no +0.10 bonus, no OCR_HINTS lookup; (2) **Full 7-scale `detector_scales` range** used for all templates, not the tight 3-scale text path — handles templates captured at non-standard scales; (3) **Canny edge channel** only for non-text templates (`0.62 × gray + 0.28 × edges`); text templates (all current keys) run **gray-only** even in Custom mode, because text edges are noisy anti-aliasing and the gray channel matches text more reliably — this ~halves Custom's matchTemplate count (e.g. full-screen 4K ~3050ms → ~1150ms) with no accuracy cost on text. `detector_force_edges: true` restores the edge channel for all templates if a custom non-text/structural template needs it; (4) **No 0.94 full-screen penalty** — user's custom template may not appear at the DEFAULT_ROI position, so the full-screen result is treated as the genuine answer. ROI is still tried first for speed, but the fallback is unpenalised.
- **ROI-only matching (`detector_roi_only`, default `true`) — the full-screen fallback is OFF by default.** Every tracked element is fixed-position and covered by its ROI (geometry-derived from the capture box, or `DEFAULT_ROIS`), so the full-screen fallback matchTemplate buys nothing and is the single most expensive per-check op. When `detector_roi_only` is on, `detect()` returns the ROI result and **never** runs the full-screen search (on either the stable `wait_for` path or single-shot `detect(stable=False)`). Measured impact at 5120×2160: **~680–740ms → ~23–35ms per detection** (~30×). This is what makes detection react promptly: the wheelspin duplicate poll runs two detections/cycle, so the old full-screen cost was ~1.5s of reaction lag per check (the "slow to act after it should have acted" report); ROI-only drops that to ~0.2s. **Kill-switch `detector_roi_only=false`** restores the ROI-first + full-screen-fallback behaviour below (for setups where an element can legitimately land outside its expected ROI — e.g. an untuned `DEFAULT_ROI` on an unusual aspect ratio). The gating below only applies when ROI-only is off.
- **Full-screen fallback gating** (only when `detector_roi_only=false`): the ROI-miss full-screen search is the single most expensive per-check op (~85% of wait-state CPU; ~525ms gray / ~3s Custom at 4K). On the stable (`wait_for`) path, `_should_run_full(key, roi_score)` skips it when the ROI score is clearly cold (`roi_score < detector_full_gate_score`, default 0.40), running it only when the ROI hints the element may have drifted (score ≥ gate) or on a periodic safety sweep (every `detector_full_sweep_every` gated checks, default 4) so drift outside the ROI is still caught within a bounded number of checks. **Single-shot `detect(stable=False)` (click ops) always runs the full fallback** — gating never causes a one-off click to miss. Conservative by design: it only skips when the ROI is cold, never trading detection for speed. Measured ~3× less wait-state CPU in Default mode (468→153ms avg). Set `detector_full_sweep_every <= 1` to disable.
- **OCR pre-warming**: `OptionalOCR._ensure_loaded()` is kicked off in a background thread when `ScreenDetector` is constructed, so the first detection call doesn't eat the 1–2s onnxruntime model-load cost.
- **OCR cooldown**: `detector_ocr_cooldown` (default 1.0s) — minimum gap between actual OCR calls per template key. Prevents CPU spikes during long failed-detection streaks. Under OCR-primary, the confirmation cache uses this same duration.
- **Warning log surfaces OCR text**: when detection stalls for 3s, the warning includes `OCR: 'xxx'` showing what OCR read in the borderline zone — critical for diagnosing why a template isn't matching on a given user's machine.
- Soft threshold: actual cutoff is `max(self._min_thresh, threshold * 0.92)` — user's slider is a target, not a hard wall. `_min_thresh` is a **hard floor** (config `detector_min_threshold`, default **0.70**) below which no match ever fires regardless of the slider: matches in the 45–70% band were misfiring on real hardware (Ally X), so the floor was raised 0.45 → 0.70. (OCR veto ceiling 0.40 stays below the floor, so vetoed matches still reject.)
- Stability filter: requires N consecutive frames (default 2) above the soft threshold; `stable=False` flag bypasses this for single-shot click ops. **`wait_for` calls `_reset_match_state(key)` at entry** — drops the per-key stability history + OCR-confirm cache so each wait needs genuinely consecutive frames. Without this, the history persisted across loops: loop 1 (empty) was correct, but loops 2+ started with the prior loop's high scores still in history, so a single transient/look-alike frame satisfied "N consecutive" and **fired early** (then desynced the flow → next step's detection "failed"). This is why the SAME loop behaved differently after several iterations. Detection must be stateless per wait.
- Optional OCR: lazy-loads `rapidocr_onnxruntime` → `rapidocr` → `pytesseract`. Hint words in `OCR_HINTS` are multilingual (en/zh-tw/zh-cn). If no OCR backend is installed, detection still works
- `ScreenDetector.wait_for(frame_cb, key, template, threshold, stop_cb, interval, timeout, on_warn)` is the loop used by race.py / mastery.py; default `timeout=float('inf')`

### Automation modules (race.py, mastery.py, delete_cars.py, buy.py, wheelspin.py)
- Each has a `run(cfg, stop_event, log_cb, status_cb, ...)` function
- Run in a daemon thread; `stop_event` signals shutdown
- **CJK IME / input handling — two-part design (all 4 scripts)**. A Traditional Chinese IME in native mode can consume keystrokes for composition *before the game reads them* (worst for Buy: Space=convert, Down=next candidate, Enter=confirm are exactly the IME candidate keys). History: v1.3.2 shipped scancode-*only* injection + an *unconditional* IME switch; both regressed users (held W not pressing on VK-path games; the forced switch clobbering users who'd already set English). v1.3.3 fixes both:
  - **(1) Dual key injection (`_send_vk`)**: sends **both** `wVk` (virtual key) **and** `wScan` (scancode via `MapVirtualKeyW`) in one event, **no** `KEYEVENTF_SCANCODE` flag. This feeds both the Windows message / VK path (`WM_KEYDOWN`/`GetAsyncKeyState`) *and* the DirectInput / Raw Input scancode path — strictly more compatible than VK-only (`wScan=0` → DirectInput gets scancode 0) or scancode-only (`wVk=0` → regresses VK-path games, notably the held `W`). Coverage: race.py + mastery.py have their own `_send_vk`; **delete_cars.py** has the extended-aware version and exposes `key_press` (public alias) the Buy worker imports. **Extended keys** (`_EXTENDED_VKS`: arrows, PgUp/PgDn/Home/End, Ins/Del, R-Ctrl/Alt) set `KEYEVENTF_EXTENDEDKEY` or they collide with the numpad block (Down `0x50` == numpad-2); delete_cars's `down` relies on this. Mastery's WASD nav uses `pydirectinput` (scancode), unchanged.
  - **(2) `capture.force_english_ime()` — conditional, non-destructive**: at run start it checks the foreground window's current input language via `GetKeyboardLayout(GetWindowThreadProcessId(hwnd))`; **if already English (LANGID primary `0x09`) it does nothing** and returns — never disturbing a user who manages their IME manually. Only when a non-English layout is active does it `LoadKeyboardLayoutW("00000409")` + `PostMessageW(WM_INPUTLANGCHANGEREQUEST)` (HKL as `c_void_p` so the 64-bit handle isn't truncated). Gated by config `auto_english_ime` (default true) at each call site — set false to skip entirely (for setups where "English" is a CJK IME in alphanumeric mode, which still reports a Chinese LANGID). Best-effort/non-fatal.
  - Mouse input is unaffected (IME is keyboard-only).
- race.py / mastery.py instantiate `ScreenDetector(cfg)` per run and call its `wait_for` / `detect`
- **Race count (race.py `max_loops`)**: the Race Auto tab has a "Number of races to run" input (`_race_count_var`, default 0 = unlimited), mirroring the mastery/buy/delete count inputs (not persisted to config). `_start_race` reads it and passes `max_loops` to `race.run`. After each completed race (W released at `restart_menu`) the loop breaks **before** pressing Restart when `loop_count >= max_loops`, so the run stops cleanly on the results/restart screen rather than kicking off a race it would immediately abandon. Logs `log_race_started_count` at start and `log_race_limit_reached` on stop.
- **Optional menu-navigation prelude (race.py) — main menu → Start screen**: before the AFK loop, race can auto-walk the menus to the EventLab event the user grinds, so other functions can eventually chain into racing. **Auto start-state detection** (no toggle): at run start, if the nav templates are loaded, it polls (~8s) for `creative_hub` OR `start_menu` — `creative_hub` (at the main menu) → run the navigation; `start_menu` (already at the Start screen) → skip it; neither → fall through to the loop's normal `wait_for(start_menu)`. **Navigation steps** (`_navigate_to_event`): click `creative_hub` → (detect+click) `eventlab` → `play_event` → `events_arrow` (the ◀ that switches the Events browser to MY HISTORY) → (detect+Enter, all pre-highlighted defaults) `my_history` (latest-played map) → `choose_race_type` (Solo) → `car_select` (already-driven car) → lands on the Start screen, where the existing loop takes over. Clicks use the shared `capture.mouse_click` at `MatchResult.location`+monitor offset; Enters use `key_press`. Each step is detection-gated and **TIME-BOXED (`_NAV_STEP_WINDOW` 12s)** — unlike the in-race loop's infinite `wait_for`, a wrong/stuck menu state won't self-correct, so a step whose element never appears **aborts** the run with `log_race_nav_fail` rather than hanging. Each step detects a *distinct* screen, so there's no lingering/re-press issue. **Best-effort / backward-compatible**: the 7 nav templates (`creative_hub`/`eventlab`/`play_event`/`events_arrow`/`my_history`/`choose_race_type`/`car_select`, all in `RACE_TEMPLATE_KEYS` + `DEFAULT_ROIS`/`OCR_HINTS`/`thresh_*`) are loaded **optionally**; missing **any** of them sets `nav_enabled=False` and race behaves exactly as before (just waits for the Start screen). So existing race setups are unaffected until they capture the nav set.
- **Race exit → main menu (race.py `_return_to_menu`) — the chaining hand-off**: when the race count is reached (the clean `max_loops` exit, where we KNOW we're on the results/`restart_menu` screen), instead of just stopping there the run can walk back to the **main menu** so every function ends on a common hand-off point. Flow: press **Enter (Continue)** to leave the race → wait through the ~12s loading screen for the recommended **"What's Next" / 接下來做什麼** menu (`next_activity`, `_EXIT_MENU_WINDOW` 30s) → **Esc** (closes it → open world) → **Esc** (→ main menu) → confirm `creative_hub` (`_EXIT_CONFIRM_WINDOW` 12s). Why capture the `next_activity` heading (fixed top-left) rather than a blind ESC-until-detected loop: it's an invariant anchor that confirms the recommended menu is up before the first ESC (the menu's *content* varies per run). **Template-gated & best-effort**: `exit_enabled = next_activity captured AND creative_hub captured` (creative_hub is the main-menu marker); missing either → the run stops on the results screen exactly as before. Steps are detection-gated + **time-boxed** (abort/log, don't hang — the run is already finishing); each step is stop-aware. Only fires on the count-reached exit — **never** on an F9/Stop mid-race (state is unknown then). `next_activity` is in `RACE_TEMPLATE_KEYS` + `DEFAULT_ROIS`/`OCR_HINTS`/`thresh_next_activity` (0.60); loaded best-effort (not in the required `TEMPLATE_KEYS`). Log strings `log_race_exit_*` / `log_race_at_menu`, bilingual.
- **Two-detection flow (race.py)**: Race Auto detects two screens — the Start Race screen and the race-END screen — with the rest timed. Per loop: (1) **`wait_for(start_menu)`** (DETECTED, not timed — the load time after a restart varies, so a fixed wait misfired; on loop 1 the user is already here so it detects immediately), (2) press Enter (start), (3) wait `_PRE_W_WAIT` (4.0s fixed) then hold W, (4) hold W until **`restart_menu` is detected**, (5) release W + press X, (6) wait `_CONFIRM_WAIT` (1.25s fixed) then press Enter (confirm). `racing`/`confirm` templates are unused; `TEMPLATE_KEYS = ["start_menu", "restart_menu"]`, and the race Setup panel (`RACE_TEMPLATE_KEYS`) shows those two rows → `thresh_start_menu` + `thresh_restart_menu` remain (`thresh_racing`/`thresh_confirm` dropped from DEFAULTS). The restart_menu `wait_for` uses `warn=False` (a race lasts minutes — a long wait is normal); start_menu keeps the default warning. Fixed waits are stop-aware (`wait()` 0.1s slices); per-step `announce()` (log+status) keeps the overlay tracking the live step. The loop-2+ "fires early on the loading screen" false-match for start_menu is still guarded by the centred-box ROI (`_ROI_BOX_KEYS`) + `_reset_match_state`. (An earlier variant timed the start step via `race_reload_wait`, but variable reload time made detection necessary — that Setting was removed.)
- **Holding W (race.py)** uses **`_send_scancode` (flagged `KEYEVENTF_SCANCODE`, `wVk=0`)**, NOT the dual `_send_vk`. Gameplay/driving input is read via the DirectInput / Raw Input **scancode** path — the dual VK+scancode (no flag) registers for menu taps but **not as a true held key for driving** (the car coasts/taps). `_send_scancode` is the same path `pydirectinput` uses for mastery's WASD nav, which works in-game. A background daemon thread re-asserts `_send_scancode('w', key_up=False)` every ~30ms while `wait_for("restart_menu")` runs; on detection/stop a `threading.Event` clears, the thread joins, and the release runs. Don't switch W back to `_send_vk`/dual — it taps.
- **W release is focus-aware (`_release_w` + `capture.focus_window`)**: injected input (SendInput) goes to whatever window currently has focus. Stopping a race via the **FAFE UI Stop button** moves focus to FAFE, so a W-up sent afterwards lands on FAFE and the **game keeps the throttle held** (stuck W) — whereas the **F9 hotkey** leaves the game focused so the release reaches it. That's why "stop via button left W held, stop via hotkey didn't." Fix: at hold start we snapshot the foreground (game) window via `get_foreground_window()`; `_release_w()` calls `focus_window(game_hwnd)` to refocus the game **before** sending `key_up('w')` + `_send_scancode('w', key_up=True)`. `focus_window` is a no-op when the game is already foreground (hotkey/normal-detection path), and uses `AttachThreadInput` to beat Windows' foreground-lock otherwise. Used on both the per-loop release and the final safety release.
- **UIPI / run-as-admin (input ignored entirely)**: if the game runs **elevated** (admin) and FAFE doesn't, Windows UIPI silently drops FAFE's injected input — the car won't move / clicks do nothing — even though global key *state* still updates (so the user "sees Windows detect the key" but the game doesn't act). This is NOT an input-method bug (scancode vs VK both fail); attempting to "fix" it by changing the injection path is wrong (and reintroduces the W-tapping regression). The fix is **running FAFE as administrator**. Documented for users in index.html Part 6 Q&A + appended to the in-app `log_warn_not_detected` stall warning, bilingual (zh-tw/en). We deliberately do **not** ship a UAC `requireAdministrator` manifest — auto-elevation is a major AV/Defender false-positive trigger (same reason the self-updater was removed).
- Per-template thresholds stored in config (keys like `thresh_start_menu`, `thresh_racing`, etc.)
- After 3s without detection: fires `warn_cb` with a red warning message that includes `best.source` and `best.score` for diagnostics. The local `wait_for(..., warn=True)` in race.py/mastery.py takes a `warn` flag — pass `warn=False` (→ `on_warn=None`) for steps where a long wait is NORMAL so the red warning is suppressed (detection itself is unchanged). Currently disabled for: race **restart_menu** (only appears after the whole race finishes), mastery **mastery_esc_hint** and **mastery_my_cars**.
- wait_for() waits indefinitely (no timeout) until detected or stopped — **intentional design**, F9 is the recovery path

#### wheelspin.py — Auto Spin Wheel

New strings `log_spin_select_mh_tab`/`log_spin_mh_tab_off`/`log_spin_end_esc`/`log_spin_back_menu`.

The click target is `MatchResult.location` (frame-absolute centre of the match) + monitor offset. If the tile isn't found within `SUPER_FIND_WINDOW` (12s) the run aborts with `log_spin_super_not_found` (a failure to START is meaningful, unlike the duplicate check). **FAFE never presses to spin**: spin #1 is the click, and every later spin auto-starts from the previous collect ("Collect Prize and **Spin Again**"). (Pressing Enter to spin landed on an already-spinning wheel and desynced after a duplicate — fixed by removing the spin trigger.)
- **Best-effort skip-forward, then collect.** A Super Wheelspin has 3 wheels and **ONE Enter collects all three prizes at once** (then either the next spin auto-starts or a duplicate menu appears), so collect is pressed exactly ONCE per spin. Per loop iteration (= one Super Wheelspin): **(1) best-effort skip** — the brief "Skip" prompt (`wheelspin_skip`) appears at the START of the reveal and the collect prompt at the END, never together, so `_wait_either(skip, collect)` waits for whichever shows first: skip → Enter (fast-forwards the reveal, lands on the collect prompt) → `_wait_gone(skip)` → `_wait_stage(collect)`; collect → fall straight through. A missed/flaky skip (or a missing skip template) just means the reveal plays out and we collect normally — skip can only *save* time, never desync (skip and collect are temporally separate, so the Enter can't hit the wrong prompt). **Skip is OPTIONAL**: `skip_tpl` is loaded only if the template exists; absent → skip disabled (`log_spin_skip_off`), run continues. **(2)** Enter (collects all 3) → **`_wait_gone(collect)`** so the lingering just-pressed prompt isn't re-detected/double-pressed. **(3)** duplicate resolution (`_wait_dup_or_next` — dup menu, OR the **next spin's skip/collect**, see below). Reuses `delete_cars.key_press` (IME-safe dual VK+scancode; only `enter`/`down`) plus the shared `capture.mouse_click`. Stop/F9 checked between every press AND inside the inner loop. **History**: skip was dropped once because ~680ms full-screen detection couldn't catch the brief (~0.5s) skip window; re-added once ROI-only detection made checks ~23ms (≈10 checks fit the window). Re-adding it also required the duplicate loop to break on the next spin's *skip* (not just collect) — see below — else the next spin's reveal would be waited out and skip would never apply past spin #1.
- **Detection helpers**: (1) `_wait_stage` (collect) uses `detector.wait_for` — waits **until** the prize/collect screen is on screen (always happens; F9/Stop recovers, an over-3s wait logs a one-time `log_spin_stage_slow` hint). (2) **`_wait_either(a, b)`** — first-of-two poll, used for the best-effort skip step (skip vs collect, whichever shows first). (3) **`_wait_dup_or_next()`** — the duplicate loop's wait: returns `'dup'` (a duplicate menu) or `'next'` (the next spin's **skip OR collect** prompt, so it breaks the instant the next spin starts rather than waiting out its reveal). (4) **`_wait_gone(COLLECT)`** runs once after the collect press so the lingering just-pressed prompt isn't re-detected and double-pressed; safe because the next collect is a whole spin away. (5) `_detect(key, tpl, window_s)` (Super Wheelspin entry tile only) is **TIME-BOXED**: a None within `SUPER_FIND_WINDOW` (12s) means "not on the My Horizon menu" → abort.
- **Duplicate inner loop — detect-after-settle (`_wait_dup_or_next`)**: 0–3 duplicate menus appear in sequence (one per duplicate car). The loop **detects a duplicate → handles it → relies on the confirming key's post-key wait (`wheelspin_post_key_wait`, 0.5s) to settle** so the menu advances to the NEXT car (or dismisses) before the next poll — then it's either the next duplicate (`'dup'` → handle) or the next spin starting (`'next'` = its skip/collect prompt → done; breaking on *skip* preserves that spin's skip window for the top-of-loop step 1). The settle (0.5s) exceeds the measured ~0.23s menu-advance, so the just-handled menu has moved on and can't be re-counted; and because the **header** template is precise it won't false-match the spinning wheel after a single duplicate. Handling per menu: **garage** → Enter; **sell** → Down×2 → Enter (the cursor resets to the top option each menu, so counts are constant). **Why this and not the alternatives** (history of three failed approaches): (a) `_wait_gone(dup)` between dupes *overshot* — it ran after the post-key wait, by which time the next menu was already up, then sat waiting for *that* to vanish until timeout (stuck on stacked dupes); (b) plain re-polling with the *body* template *re-counted* — the body text false-matched the dismissing menu / wheel at ~85%, **selling one duplicate three times**; (c) **rising-edge** counting (count on absent→present) got **stuck on dup #2** — the "Car Already Owned" header is so stable it stays detected *continuously* from one duplicate into the next (no absent gap), so the edge never fires. The current approach sidesteps all three by leaning on the header's precision/reliability plus a settle longer than the advance. **Timing caveat:** correctness depends on settle (0.5s) > menu-advance (~0.23s); validated on the real game, not stubs (stubs can't model the real-clock settle). Verified with stubs only that the *continuous-header* case handles 1/2/3 duplicates without sticking. Up to `MAX_DUP_CHAIN` (**3** — a Super Wheelspin has 3 prizes). **Detection *reliability* — capture the header, not the body:** an earlier `wheelspin_duplicate` capture was the *"What would you like to do with this car?"* body text, which sits over the (car-dependent) menu art, so its match swung by car (~85% when it hit, sometimes below the 0.60 threshold → menu on screen but never detected, loop waits forever showing "waiting for duplicate or next spin"). Fixed by **capturing the duplicate template on the solid-yellow "Car Already Owned" header** (invariant background → consistent match; the phrase is also unique to that menu). `OCR_HINTS["wheelspin_duplicate"]` leads with `"already owned"/"owned"` to match it (old body hints kept as fallback). NOTE: a capture must be on the **header**, not the body, or this regresses. (The EN set is on the header; if a game-language set is recaptured, do the same.) **Detailed diagnostic log** (the wheelspin log is for diagnosis): each step logs what it's doing — `→ waiting for collect prompt`, `✓ {label} detected ({conf}) — {secs}s` (`log_spin_detected`, with **elapsed time** so reaction latency is visible), `→ collect: Enter (takes all 3 prizes)`, `→ waiting for duplicate menu or next spin`, `→ duplicate #n: Sell (Down×2 → Enter)` / `Add to Garage (Enter)`, `✓ no more duplicates — next spin ({secs}s)`. The `conf%`+`source` exposes misfires (low %, or a `full` source); the `secs` exposes slow detection. The status bar tracks the current `→` action (which also fixed the old stale "Collecting reward…" status that lingered while the next wheel was already spinning). Mode = `wheelspin_dup_mode` (`"garage"` default | `"sell"`), a CTkSegmentedButton on the tab. **Sell sells duplicates automatically + unattended** — warned explicitly in the bilingual `spin_description` + a startup `log_spin_sell_warn`.
- **FOUR detect templates** (`SPIN_TEMPLATE_KEYS`, captured in the Setup panel; all in `DEFAULT_ROIS` + `OCR_HINTS`, so all text templates; `thresh_<key>` 0.60 each): `super_wheelspin` (left-column tile, left band, clicked at `MatchResult.location`), `wheelspin_skip` (the **bottom-left "Skip" prompt** during the reveal — white text on the solid teal bar, stable bg; used for the **best-effort** fast-forward), `wheelspin_collect` (the **bottom-left button prompt** at the prize — `(0.00, 0.86, 0.40, 0.14)`, captured as the prompt TEXT, far more stable than the animating prize art), `wheelspin_duplicate` (the solid-yellow **"Car Already Owned" header**). `wheelspin_skip` is best-effort: `wheelspin.py` loads it only if present (`skip_tpl=None` → skip disabled, run continues), so a missing skip capture never blocks a run even though it's in the Setup list. ### Capture (capture.py)
- `CaptureSession`: listens for CAPS LOCK, opens fullscreen region selector, shows preview
- `NodeSession`: waits for CAPS LOCK, opens full screenshot for clicking 6 node positions
- ESC cancels at any stage — `_on_esc` also fires `on_cancel()` when not mid-capture (not just mid-capture)
- **Multi-capture fix**: `self._stop.set()` is called immediately before `self.callback()` in the success path, ensuring the old session's CAPS LOCK listener is torn down before the next session registers its own. This prevents duplicate listeners that previously broke the 2nd/3rd capture.
- Hooks are targeted (not unhook_all) to avoid killing the F9 toggle hotkey
- **Template rescaling**: `load_template` scales by height only (`scale = current_h / meta["screen_height"]`). Game UIs are laid out relative to vertical resolution; using an average of width+height was wrong for non-standard aspect ratios (e.g. 5K2K).
- **Single-reference templates (Stage 1 — resolution borrow from ok-script)**: built-in templates are authored once at `config.REFERENCE_RES` (`"2160p"`, the highest) and **downscaled** to lower resolutions at load time — downscaling preserves text/edges far better than upscaling a low-res crop, so one set serves every preset resolution. `load_template(..., ref_folder=None, prefer_ref=False)` picks the source: `prefer_ref` + ref-has-it → ref (downscale); else the per-res `folder`; else fall back to ref. Driven by config `template_prefer_reference` (default true) — a reversible kill-switch: set false to use each resolution's own folder again. race.py/wheelspin.py pass `ref_folder = get_*_templates(REFERENCE_RES, lang)` + the flag for **preset** resolutions only; **custom** never falls back (it's the user's own capture). Defaults `ref_folder=None, prefer_ref=False` reproduce the old behaviour exactly. The Setup panel resolution picker is collapsed to **Built-in (auto-scaled) / Custom** (`setup_panel._build_content`): Built-in maps to `REFERENCE_RES`; the old 1080p/1440p/2160p numbers are gone (meaningless once one set serves all). Any legacy stored preset still displays as Built-in and behaves identically. **STATUS**: the 1080p/1440p folders are still shipped (instant revert via the flag) and match-quality of the downscaled 2160p set at native 1080p/1440p is **pending real-game validation** before they're deleted. **Stage 2 (built)**: templates now carry their **capture box geometry**. `save_template(..., box=(x,y,w,h))` writes it to the sidecar JSON (threaded from `CaptureSession` → setup_panel `_on_captured`); `load_template` returns `(img, scale, meta)`. When a template has a `box`, the caller calls `detector.set_template_geometry(key, box, cap_w, cap_h)` and `detect()` derives an **anchor-aware ROI** from it (`_geom_roi`), then pads by `detector_geom_variance` (default 0.05). **Two anchoring models, chosen per key:** **(menus, the default)** the box is mapped through the **centred 16:9 content box** of BOTH the capture and the live frame (`_content_box`, the same model `capture.load_nodes` uses) — the element's fraction *within* the content box is preserved, so the pillarbox/letterbox offset is handled across aspect changes. **(edge HUD, keys in `_GEOM_EDGE_KEYS` — currently only `racing`, the in-race timer)** scale by height + `_scale_by_anchor` to the nearest screen edge. **WHY the content-box model:** templates captured on the dev's **5120×2160 ultrawide** (where Forza pillarboxes menus ~640px into a centred 16:9 box) were failing to detect on **16:9 targets** (Ally X 1920×1080, and any 16:9 user) — the old pure edge-anchor model carried the pillarbox offset into x and mis-scaled it (`x=821 → ROI ~410` but the element is at `~90`), landing the ROI ~320px off; with `detector_roi_only=true` (no full-screen fallback) that = hard detection failure on the first step. The content-box model fixes it (`x → 90`, element inside ROI) on every aspect, **with no same-aspect regression** (capture aspect == live aspect → identical ROIs). This **replaces the hand-tuned `DEFAULT_ROIS` for geometry-bearing templates only** — the no-`box` set still uses `DEFAULT_ROIS`. Kill-switch `detector_geometry_roi=false`. The geometry ROI bypasses the 16:9 `_roi_for_frame` remap (it accounts for aspect itself). Verified numerically: start_menu/creative_hub/restart_menu land inside the ROI on 5120×2160 / 1920×1080 / 3840×2160 / 2560×1440, and the 5K2K ROIs are byte-identical to the old edge model.
- **Node click rescaling (aspect-aware)**: `load_nodes(file, w, h, aspect_fix=True)` maps stored `x_ratio`/`y_ratio` through a **centred 16:9 content box** (`_content_box`, fit-contain) instead of naive `ratio × width`. Forza anchors menu UI to a centred 16:9 region (pillarboxed on ultrawide, letterboxed on tall screens), not stretched — so a 16:9 preset node at x_ratio 0.194 must land in that centred box on a 21:9 screen, not at 0.194×width. Verified against a real 5120×2160 capture: the 16:9 preset remaps to within 0–15px of the user's hand-captured nodes (well inside click tolerance), and same-aspect captures map to themselves (≤1px round-off). Reduces to naive scaling when capture & current aspect match, so 16:9 users are unaffected. Toggle via `nodes_aspect_fix` (config, default true). The model was confirmed empirically: 16:9 ratio 0.194 → predicted 0.271 → measured 0.2707 on 5K2K (centred-box, not naive=0.194 or left-anchored=0.145).
- **Frame capture reuse + periodic refresh**: `grab_frame` uses a thread-local mss instance via `_get_sct()` (stored in `threading.local()` as `_tls.sct`), created lazily and reused, **and rebuilt every `_SCT_REFRESH_EVERY` (400) grabs** (`_tls.sct_count`). Reuse fixes the per-call GDI leak; the periodic rebuild flushes any *native* GDI/handle accumulation that some mss versions still leak slowly even on a reused instance over thousands of grabs (the leftover cause of long runs deteriorating after the per-call fix — the Python/detector side is provably flat via tracemalloc, so the residual is native/mss). Previously it built/destroyed `mss.MSS()` on every call — the detection loop grabs ~2×/sec indefinitely, so over a long run that leaked Windows GDI device-contexts/bitmaps until `grab()` returned black frames, which is why race/mastery grew more failure-prone the longer they ran. An mss instance is bound to its creating thread, hence thread-local rather than a global singleton. `grab_frame` has a one-time rebuild fallback: a non-`RuntimeError` exception (e.g. display/DC reset) closes the stale instance, recreates it once, and retries; `RuntimeError` (monitor-not-found / no-image) is re-raised. `list_monitors` / `get_monitor_dims` keep the one-shot `with mss.MSS()` — called rarely (startup) and possibly from the UI thread.

### Config (config.py)
- JSON file next to exe
- **Self-completing on load**: `load()` back-fills any keys missing from the user's `config.json` with `DEFAULTS` **and writes them back to disk** (once — idempotent, only when something was added). Since the updater leaves `config.json` untouched, this is how keys added in a new version (e.g. `auto_english_ime`, `ui_scale`, `nodes_aspect_fix`) appear in an existing user's file so they can see/toggle them. Existing user values are preserved; a corrupt file is left as-is (returns defaults without overwriting); a missing file is created from `DEFAULTS`. **`DEFAULTS` must list every user-facing key for this to surface it** — the optional `detector_*` deep-tuning keys are intentionally NOT in `DEFAULTS` (they'd clutter the file) and remain code-default overrides for advanced users.
- Keys include: lang, theme, toggle_key, capture_key, report_key, overlay_key, monitor_index, race_resolution, mastery_resolution, all thresh_* values, all *_wait values
- Mastery-specific timing keys: `mastery_check_interval` (min 0.3), `mastery_post_click_wait` (min 0.5), `mastery_post_key_wait` (min 0.8), `mastery_node_click_wait` (min 0.7)
- `mastery_start_loop` (int 1–3): which row the first car is on for Auto Unlock 22B
- Mastery is keyboard-driven only (no `mastery_mode`, no detection). Its step waits are fixed constants in mastery.py (see mastery.py → keyboard-driven flow); only `mastery_node_click_wait` + `mastery_start_loop` are configurable.
- `nodes_aspect_fix` (bool, default true): aspect-aware mastery node click remapping via the centred 16:9 content box (see Capture → Node click rescaling)
- `ui_scale` (str, default `"auto"`): UI scale factor — `"auto"` (resolution-derived) or a percentage like `"150%"`. Resolved by `config.resolve_ui_scale`; applied via CTk widget/window scaling (see UI → UI scale)
- `overlay_enabled` (bool, default false): show the translucent game-status overlay over a borderless game while a script runs (see Game status overlay)
- `auto_english_ime` (bool, default true): at run start, switch the game window to English input **only if** a non-English layout is active (no-op when already English). Set false to never touch the IME — for users who manage it manually, esp. where "English" is a CJK IME in alphanumeric mode (still reports a Chinese LANGID, so the auto-skip wouldn't catch it). See Automation modules → CJK IME.
- Optional detector tuning keys: `detector_roi_only` (bool, default **true** — ROI-only matching; skip the full-screen fallback entirely since elements are fixed-position. ~30× faster per detection; set false to restore the ROI-first + full-fallback behaviour), `detector_scales` (list of floats), `detector_text_scales` (list of floats), `detector_stable_frames` (int), `detector_enable_ocr` (bool), `detector_ocr_cooldown` (float seconds, default 1.0), `detector_ocr_primary` (bool, default **true** — OCR as first-class confirm+veto signal; **auto-set per run** from the template type: preset resolution → true, custom → false; no manual UI toggle), `detector_ocr_skip_above` (float, default **0.90** — pixel score at/above which OCR is skipped and the pixel match trusted; raised from 0.65→0.90 so coincidental ~0.80 pixel matches fall into the OCR-decided band instead of being trusted), `detector_ocr_skip_below` (float, default 0.20 — below this, no match and OCR skipped), `detector_ocr_veto` (bool, default **false** — opt-in; when true, reject a match if OCR positively reads non-hint text. Default off because a hard veto stalls detection on machines whose OCR can't read the game text), `detector_ocr_veto_ceiling` (float, default 0.40 — when the veto fires, the score is capped to this, below the 0.45 soft-threshold floor), `detector_ocr_confirm_score` (float, default 0.85), `detector_ocr_cache_pixel_min` (float, default 0.15 — minimum pixel score for cached OCR confirmation to apply), `detector_full_gate_score` (float, default 0.40 — ROI score at/above which the full-screen fallback always runs), `detector_full_sweep_every` (int, default 4 — run the full-screen fallback every Nth gated check as a safety sweep; ≤1 disables gating), `detector_force_edges` (bool, default false — force Canny edge channel on for all templates incl. text), `detector_roi_aspect_fix` (bool, default true — expand ROIs along the distorted axis on non-16:9 screens), `detector_roi_aspect_tol` (float, default 0.05 — ± fraction of 16:9 treated as still-16:9)

### Log widget (log_widget.py)
- Thread-safe via queue
- `log(msg)` — normal text
- `log_warning(msg)` — red text (#ff4444) via tk.Text tag
- `log_section(msg)` — starts a new bounded loop section with `msg` as its header. Only the last `MAX_SECTIONS` (default 3) sections are retained; opening the 4th deletes the oldest from the top of both `self._lines` and the tk widget so the log doesn't grow unbounded across long runs. A blank separator line is inserted between sections (counts into the new section). Lines logged before the first `log_section` form an implicit "preamble" section (startup/template-load messages), trimmed once 3 loop sections exist.
- Section bookkeeping: `self._section_lens` tracks line count per live section (front = oldest). The hard `MAX_LINES` (1000) safety cap still applies and decrements `_section_lens` from the front to stay in sync. **Section headers must not contain embedded newlines** — the widget assumes one line per buffer entry, so a `\n` in a header would desync line accounting (this is why race.py's `-- Loop #N --` header dropped its leading `\n`).
- Wiring: `race.run` / `mastery.run` take an optional `section_cb` (falls back to `log_cb`); `main_window` passes `self._race_log.log_section` / `self._mastery_log.log_section`. Loop/car headers go through it. Buy/Delete tabs don't pass `section_cb`, so their logs are unsectioned (single growing preamble).

### Game status overlay (overlay.py)
- `GameOverlay` is a borderless, **topmost**, **opaque** `tk.Toplevel` styled as a small card: a 1px border, a **header bar** (FAFE icon via `PIL.ImageTk` from `icon_path` + "FAFE" text, plus a **function-switcher dropdown** on the right — `tk.OptionMenu` of `tabs=[(key,label)]`, `on_tab(key)`→`_switch_tab`, kept in sync via `set_current_tab(self._current_tab)` each refresh; short labels `ov_func_*`), the current **status** (green), the **last 5 active-log lines** (mono), and a **footer hint** showing the start/stop key (`overlay_hint_start`/`overlay_hint_stop`, e.g. "■ Press F9 to stop", chosen by running state). Updated via `update(status, lines, hint)`. Logo needs `--hidden-import PIL.ImageTk` in the build (else it silently falls back to no logo). Toggle via the Settings → Appearance switch (`overlay_enabled`, default false, label `setting_overlay`) **or the `overlay_key` hotkey (default F10)** — both flip the same `overlay_enabled` flag and stay in sync. There's also a **clickable status indicator in the topbar** (right of the monitor picker): green ● = on, gray ○ = off; clicking it toggles. All three entry points (settings switch, F10 hotkey, topbar indicator) route through one method `_set_overlay_enabled(enabled)` which writes config + syncs the switch var + refreshes the indicator (`_update_overlay_indicator`); `_refresh_overlay` does the actual show/hide. The F10 handler (`_toggle_overlay`) runs on the keyboard thread so it marshals to the main thread via `self.after`. `overlay_key` is rebindable in Settings alongside the toggle/capture/report keys. (The indicator also makes the on/off state visible — important because F10 *toggles*, so if the overlay was already on, F10 turns it off.)
- **Works over borderless / windowed games only** — an *exclusive-fullscreen* game owns the display and nothing can draw over it. (Most automation users run borderless, which is also what the IME/window-message handling assumes.)
- **Never self-detected — frame blanking (`capture.set_overlay_mask`)** — CRITICAL: the automation must never detect its OWN overlay window. The overlay is a dark card with text, so it (a) pixel-matches the game's menu/dialog templates and (b) shows log/status text containing template **hint words** ("Confirm", "Restart", "Start") that OCR reads as false confirmations — a detection feedback loop. The `confirm`/`racing` ROIs overlap the overlay, so this caused mid-race phantom matches (a real v1.3.4 bug report: the script ran a full loop while still in the first race). **Solution = blank the overlay's own screen rect out of every detection frame.** Because the overlay is our window we know its exact rect (`GameOverlay.bounds()` → `winfo_rootx/y/width/height`, absolute screen px); `_refresh_overlay` registers it via `capture.set_overlay_mask(x,y,w,h)` (re-read each ~500ms poll + immediately on drag via `_on_overlay_move`→`_update_overlay_mask`; cleared on hide). `grab_frame(monitor_index, mask_overlay=True)` translates that rect to frame-local coords and fills it with black (a flat region can't match a text template). **Why not `SetWindowDisplayAffinity(WDA_EXCLUDEFROMCAPTURE)`** (the first attempt): on some displays — notably handhelds like the **Xbox/ROG Ally**, whose built-in screen is driven through a capture/composition path — exclude-from-capture hides the window **from the user too**, and there's no reliable way to detect that, so it can't guarantee the requirement. Blanking works on every display path and keeps the overlay fully visible. **Trade-off**: the detector can't see game UI *directly under* the overlay, so the default overlay position is the **top-right corner** (clear of all 16:9 detection ROIs) and the user drags it elsewhere as needed (position persists). Bug-report capture uses `mask_overlay=False` so the report screenshot shows the real screen, not a black box. (For the rarer game-scene pixel coincidence unrelated to the overlay, the opt-in `detector_ocr_veto` exists — see Default mode.)
- **Draggable, focus-safe, OPAQUE.** Only `WS_EX_NOACTIVATE` (never takes focus from the game → input keeps working), applied via ctypes while hidden so the first show never steals focus. **No `-alpha`/`WS_EX_LAYERED`** (translucency) and **no `WS_EX_TRANSPARENT`** (click-through): both layered-alpha *and* transparent windows composited the window background but **not the child labels** on the test setup → a featureless **black box**. Opaque guarantees the labels render; not click-through allows dragging (user drags it off click targets). Translucency could be re-attempted later (e.g. re-assert `SetLayeredWindowAttributes` after the `SetWindowLong`), but opaque is the reliable baseline. Trade-off: it's no longer click-through, so it could intercept an injected click if parked over a target — the user drags it out of the way (the whole box is a drag handle, cursor `fleur`, with a `⠿ FAFE` hint header). Position persists via `overlay_x`/`overlay_y` (saved on drag-end through the `on_move` callback → `_on_overlay_move`); `_refresh_overlay` uses the saved position if set, else the game monitor's top-left.
- **`show()` must set an explicit `WxH` geometry**, not just `+x+y`: an `overrideredirect` (borderless) Toplevel given only a position maps at **1×1 (invisible)** — this was the "indicator green but no window, no error" bug. `show()` deiconifies, `update_idletasks()`, then sets `{_WIDTH}x{reqheight}+{x}+{y}`.
- **Log label is fixed at `height=5` text lines** so the window is sized for 5 lines from the first `show()`. Without it the window sizes to the initial (few) lines and later/**newest** entries get clipped off the bottom (the window sizes once, on first show). `update(status, lines, hint)` is passed `_lines[-5:]`, newest at the bottom.
- `MainWindow._refresh_overlay()` polls ~2×/sec: when `overlay_enabled`, it lazily creates the overlay, positions it once on the game monitor's top-left (`get_monitor_dims` + 24px margin, cached in `_overlay_pos`, recomputed after each hide), pushes status + last 5 active-log lines, and `show()`s; otherwise `hide()`s. (Shows whenever enabled — not gated to a running script — so the F10 toggle gives immediate visible feedback.) Plain `tk` (not CTk) to avoid the CTk window-scaling factor distorting the physical-pixel monitor positioning.

### Bug report (report.py)
- **One-press diagnostic bundle**, triggered by a global hotkey (`report_key`, default **F12**, rebindable in Settings alongside toggle/capture keys). Registered in `MainWindow.__init__` via `keyboard.add_hotkey`; live re-registration on rebind in `_on_key_bind_press` (remove old + add new).
- `MainWindow._trigger_report()` snapshots each tab's log lines on the UI thread, then runs `report.generate_report(cfg, monitor_index, logs, log_cb, status_cb)` on a **daemon thread** (DxDiag is slow). Guarded by `self._report_in_progress` against double-fire.
- `report.generate_report` writes `reports/FAFE_report_<ts>/` with: **`screenshot.png`** (current monitor via `grab_frame`, written with `cv2.imencode`+file-write so non-ASCII/CJK paths work — *not* `cv2.imwrite`), **`log.txt`** (version + `platform` env + full config dump + all tab logs), **`dxdiag.txt`** (`dxdiag /t` with `CREATE_NO_WINDOW`, 120s timeout, basic `platform` fallback if it fails), then zips the folder → `FAFE_report_<ts>.zip` and opens the folder via `os.startfile`. Status/log messages are localized (`report_generating`/`report_specs`/`report_saved`). After saving, a localized **privacy notice** (`report_privacy`) is logged and a matching header line is written into `log.txt`, so the user knows the bundle contains their settings + system info (DxDiag includes username/hardware) before sharing. Reports save under `config.BASE_DIR/reports/` (gitignored; updater won't touch it).
- **🐞 Report a Bug button → inline tutorial, NOT a direct capture.** The bottom-right overlay has a Report button (`_open_report_help`/`_close_report_help`/`_build_report_help_panel`, mirrors the support/settings inline pattern) that explains the F12 flow in 4 steps + a privacy note + a Discord button. It does **not** trigger capture itself: clicking a button foregrounds the FAFE window and (single-monitor) would screenshot FAFE instead of the game. The actual capture only happens via the **global hotkey**, pressed while the game is in focus. Step 2 shows the live `report_key` (`_at("report_help_step2", key=...)`).

### Update check (updater.py) — CHECK-ONLY, no download/install
- **FAFE does NOT download or install updates itself.** It only does a read-only **version check** against the GitHub API; if a newer release exists, the UI shows a notice and a button that **opens the releases page in the browser** (`webbrowser.open(RELEASES_PAGE)` — a browser handoff, NOT the exe downloading). **Why removed**: (1) the old self-updater (download zip → write a `.bat` → `subprocess` it → overwrite FAFE.exe + relaunch) was the single biggest **antivirus false-positive trigger** — Windows Defender began quarantining FAFE on users' machines (`病毒或潛在的垃圾軟體`); (2) **Nexus Mods prohibits** bundled executables that download/send files over the internet ("auto-update is not crucial"). Removing it fixes both. `updater.py` no longer imports `subprocess`/`zipfile`/`tempfile`.
- `check_async(on_update_available, on_status)` runs the check in a daemon thread; `on_status(msg)` reports the outcome (`checking…` / `up to date (latest vX)` / `update available` / `update check failed: <reason>`) to the active tab log + persistent `fafe_diag.log`. `fetch_latest` returns `(tag, RELEASES_PAGE)` and records failures in `updater._last_error`.
- **`update_check` config (default true)** gates the check. The check is still an outbound HTTPS request, which Nexus's policy technically also covers, so a **fully offline / Nexus-strict build sets `update_check=false`** (zero network — the exe makes no requests at all; users update manually from the page).
- **HTTPS uses certifi's CA bundle** (`_SSL_CTX = ssl.create_default_context(cafile=certifi.where())`) so the check works on Windows installs whose OS root-cert store is incomplete (e.g. handhelds) — otherwise `CERTIFICATE_VERIFY_FAILED`. Bundled via `--collect-all certifi` + `--hidden-import certifi`.
- **`fafe_diag.log`** (`MainWindow._diag`/`_diag_init`, written to `config.BASE_DIR`): persistent diagnostics file (truncated each launch) capturing version/env, monitor enumeration, overlay create/show state, and the update-check outcome. Survives the bounded/trimmed UI log and works in `--windowed` (no stderr).
- **Distribution note**: because the app no longer self-updates, each release must be downloaded manually (GitHub releases page / Nexus). The detector/templates ship in the zip as before.

## Key Design Decisions
- All UI strings go through `_at(key, lang, **kwargs)` in app_lang.py — never hardcode UI text
- Sliders save on mouse release only (not on drag)
- Slider minimums are set to the lowest safe values — going lower breaks automation because the game can't keep up
- Per-template threshold sliders in Setup panel (custom mode only shows retake buttons)
- Settings sliders clamp loaded values to slider range to handle stale config
- `keyboard.unhook_all()` is NEVER called — only targeted hook cleanup to preserve F9
- `wait_for` is intentionally infinite with no auto-retry — retrying detection without changing template/threshold yields the same result; user adjusts via sliders or recaptures
- All templates are text-based → `TEXT_TEMPLATES` covers all keys; never add a non-text template without also removing it from `TEXT_TEMPLATES` and verifying the edge channel improves accuracy
